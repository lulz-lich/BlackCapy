Você está trabalhando no projeto BlackCapy.

BlackCapy é uma ferramenta portátil profissional baseada em ESP32, com foco em interação com hardware, análise wireless, automação, logs estruturados, microSD nativo, módulos expansores e interface física com display 320x240 e botões estilo Flipper Zero.

Filosofia do projeto:
- Inspirado em Arch Linux: liberdade, modularidade, controle e customização.
- Não deve ser nichado/difícil como Arch.
- Deve ser profissional, poderoso, acessível o suficiente para uso amplo e viável para investimento.
- Pode ser usado com intuito educacional, mas o foco do produto é profissional.
- A documentação do GitHub deve ser altamente educativa e instrutiva.

Regras importantes:
- Evitar bloat. Não adicionar ferramenta só por adicionar.
- Priorizar arquitetura, consistência e manutenção.
- AppManager é a fonte única de apps.
- Não criar ToolManager, AppManagerV2 ou sistemas duplicados.
- ModuleManager controla módulos externos.
- ModuleManifest define tipos de módulos suportados.
- CaptureWriter salva evidências operacionais no microSD.
- Logger salva eventos do sistema.
- Logger não deve substituir CaptureWriter.
- microSD é nativo/onboard, não módulo externo.
- Módulos expansores são para capacidades especializadas: RF, Sub-GHz, IR, NFC, RFID, CAN, GPS, LoRa, sensores etc.
- BadUSB não é prioridade agora.
- Sub-GHz futuro deve mirar nível Flipper Zero ou superior, não apenas captura simples.

UI:
- Interface híbrida por padrão.
- ASCII para dados, menus, logs e navegação técnica.
- Pixel art para splash, ícones, contexto visual, status e animações.
- Suportar futuramente ASCII estático, ASCII animado, pixel art estática e pixel art animada.
- Performance sempre vem antes de estética.
- Serial é interface temporária/debug, não interface final.

IA futura:

- Diagnóstico de erros.
- Sugestão de automações.
- Resumo de capturas.

- Não enfiar IA diretamente no firmware de forma acoplada.

Hardware alvo:

Arquitetura atual:
- AppManager: registra apps, seleciona app, executa app, verifica se app pode rodar.
- AppContract: define permissões e status dos apps.
- ModuleManifest: define tipos de módulos: IR, RF, RFID, NFC, CAN, GPS, LoRa.
- ModuleManager: detecta/registra módulos, verifica se módulo existe, suporta mock e ADC ID.
- CaptureWriter: salva capturas em /captures/*.log no microSD.
- FileSystem: inicializa microSD e cria /logs, /assets, /scripts, /captures, /modules.
- Storage: usa Preferences para configs críticas.
- StoragePolicy: separa Preferences para configs críticas e microSD para dados operacionais.
- Logger: logs de sistema, com opção de gravar em /logs/system.log.
- InputManager: lê botões físicos com debounce.
- DisplayManager: abstração para display 320x240, atualmente fallback serial.
- ScreenManager: renderiza Home, Tools, Status, Settings. Tools deve mostrar 4 itens por tela.
- UIController: conecta botões com telas e AppManager.
- Menu serial: usa AppManager, somente debug/interface temporária.
- EventBus: eventos como boot, tool started, tool finished, module detected, error.
- StatusManager: IDLE, RUNNING_TOOL, ERROR.
- AutomationEngine: base inicial.

Regra:
Logger = comportamento do sistema.
CaptureWriter = evidência operacional.

microSD:
- É nativo/onboard.
- Deve armazenar:
  - /logs
  - /assets
  - /scripts
  - /captures
  - /modules
  - /themes
  - /plugins futuramente

Sub-GHz futuro:
- Deve ser nível Flipper Zero ou superior.
- Não tratar como captura simples.
- Deve ter scan, frequency sweep, RSSI em tempo real, raw capture, decoder, banco de sinais no microSD, UI dedicada, integração com AppManager/ModuleManager/CaptureWriter e replay autorizado com confirmação física e logs obrigatórios.

BadUSB:
- Não é necessário agora.
- Se for feito no futuro, deve ser módulo externo com hardware USB HID nativo, como ESP32-S3/RP2040, desativado por padrão, com confirmação física e logs.

Ao modificar código:
- Não recriar módulos existentes.
- Não inventar nomes novos se já existe módulo.
- Sempre refatorar em cima da arquitetura atual.
- Se alguma função não existir, adaptar ao padrão atual em vez de criar arquitetura paralela.
- Evitar identificadores fantasmas.
- Gerar arquivos completos quando solicitado.
- Manter compatibilidade com PlatformIO.
- Considerar que o platformio.ini usa src_dir = main dentro da seção [platformio].
- Cuidado com tamanho do firmware: BLE pode exigir huge_app.csv ou NimBLE futuramente.
- Cuidado com conflitos de pino em hardware_config.h, especialmente SPI CS entre SD, RF, LoRa, CAN e RFID.
